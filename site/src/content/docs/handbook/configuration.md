## Environment variables

| Variable | Purpose |
|----------|---------|
| `GITHUB_TOKEN` | GitHub PAT for remote/crawl/publish (5000 req/hr vs 60) |
| `OLLAMA_HOST` | Override Ollama endpoint (default: auto-detect) |
| `ARTIFACT_OLLAMA_MODEL` | Force a specific Ollama model |

## Config commands

Read and write config values:

```bash
artifact config get agent_name
artifact config set agent_name vera
```

The config file lives at `~/.artifact/config.json`. The primary config key is `agent_name`, which controls the active curator persona (valid values: `glyph`, `mina`, `vera`). The default persona is `glyph`.

## Tiers

Every decision assigns one of five tiers. The inference engine computes tier weights based on the repo's archetype, bottleneck, maturity, and risk profile.

| Tier | Purpose |
|------|---------|
| **Exec** | Executive summaries, system maps, risk placards |
| **Dev** | Quickstart cards, integration recipes, debug trees |
| **Creator** | Logo variants, icon sets, sticker sheets, theme palettes |
| **Fun** | Board games, card decks, choose-your-adventure, museum placards |
| **Promotion** | One-slide pitches, demo scripts, launch post kits |

## Repo type classification

Artifact auto-detects repo type from truth atoms. You can override with `--type`:

| Type | Description |
|------|-------------|
| `R1_tooling_cli` | CLI tools |
| `R2_library_sdk` | Libraries and SDKs |
| `R3_service_server` | Services and servers |
| `R4_template_scaffold` | Templates and scaffolds |
| `R5_spec_protocol` | Specs and protocols |
| `R6_demo_showcase` | Demos and showcases |
| `R7_data_registry` | Data registries and catalogs |
| `R8_product_app` | Product applications |
| `R9_brand_meta` | Brand and meta repos |

## Output locations

### Per-repo output

Each `drive` writes to `.artifact/` in the target repo:

- `decision_packet.json` — the full decision with tier, formats, constraints, hooks, freshness payload
- `truth_bundle.json` — extracted truth atoms with `file:line` citations
- `memory.json` — per-repo memory entries for the Curator
- `ARTIFACT_BLUEPRINT.md` + `blueprint.json` — generated by `blueprint` command
- `assets/` — placeholder skeleton created by `blueprint`

### Global state

Org-level data lives at `~/.artifact/`:

- `config.json` — user config (active persona, etc.)
- `repos/` — cached remote repo data
- `org/` — season state, ledger, bans, signature moves
- `org-memory.json` — org-level memory entries (cross-repo patterns)

Per-repo memory entries are stored at `<repo>/.artifact/memory.json` alongside the decision packet.

## Decision packet structure

The core output is a JSON decision packet:

```json
{
  "repo_name": "my-tool",
  "tier": "Fun",
  "format_candidates": ["F2_card_deck", "F9_museum_placard"],
  "constraints": ["monospace-only", "uses-failure-mode"],
  "must_include": ["one real invariant", "one failure mode", "one CLI flag"],
  "ban_list": ["generic-readme-badge"],
  "freshness_payload": {
    "weird_detail": "...",
    "recent_change": "...",
    "sharp_edge": "..."
  },
  "selected_hooks": [
    { "atom_id": "invariant:abc123", "role": "invariant_hook" }
  ],
  "callouts": {
    "veto": "...",
    "twist": "...",
    "pick": "...",
    "risk": "..."
  },
  "driver_meta": {
    "host": "localhost:11434",
    "model": "llama3.2",
    "mode": "ollama",
    "timestamp": "2026-03-25T12:00:00Z"
  }
}
```

The `freshness_payload` contains three hand-picked signals from the repo's actual source code. The `selected_hooks` trace each choice to a truth atom ID. The `callouts` carry the Curator's editorial voice (veto, twist, pick, risk). The `driver_meta` records whether Ollama or fallback was used.
